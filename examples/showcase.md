# Showcase Examples

These examples are designed for public GitHub presentation. They show the kind of practical, interview-relevant engineering work this skill is meant to support.

## Example 1: Multi-role Campus System Hardening

**Prompt**

`Review this Spring Boot campus recruitment system and make it more production-ready without inventing fake complexity.`

**A strong output should include**

- clear identification of the highest-leverage weak spots
- targeted fixes in validation, error handling, API consistency, or test coverage
- explanation of what changed for end users or reviewers
- honest framing of what is still student-project scope

**Why this showcases the skill**

This demonstrates engineering judgment: improving credibility and quality without overengineering the project.

## Example 2: Recommendation Query and Endpoint Cleanup

**Prompt**

`Optimize the recommendation list query, reduce response inconsistency, and keep the API backward compatible.`

**A strong output should include**

- query or mapper analysis
- API contract review
- minimal safe changes with clear compatibility notes
- verification steps for SQL behavior and endpoint output

**Why this showcases the skill**

This example shows that the skill can connect SQL, API design, and maintainability in one coherent engineering pass.

## Example 3: Interview-ready Architecture Review

**Prompt**

`Read this repo and tell me what the strongest engineering talking points are for an interview, then improve the weakest implementation area.`

**A strong output should include**

- architecture strengths tied to real code evidence
- one meaningful implementation improvement
- language the owner can reuse in interviews
- explicit note of any unverifiable claims that should not be overstated

**Why this showcases the skill**

This makes the skill feel more than bug-fixing. It becomes a bridge between code quality and project presentation.
